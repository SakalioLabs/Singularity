# Paper Card: WISE

**Title**: WISE: A Long-Horizon Agent in Minecraft with Why-Which Reasoning
**Source**: https://arxiv.org/abs/2606.12852
**Date**: 2026
**Authors**: Renmin Cheng, Changhao Chen
**Type**: Paper
**Core Problem**: What-where-when episodic memory is not enough when low-level controllers repeatedly fail or miss causal opportunities.
**Core Method**: Which-Why Informed Semantic Explorer with causal event graph memory and opportunistic task scheduling.
**System Architecture**:
  - Causal event graph connects observations, task relevance, and outcomes.
  - Opportunistic scheduler reorders subtasks when relevant resources or events appear.
  - Progressive exploration improves observation coverage.
**Memory Mechanism**: Causal event graph over experience rather than flat episodes.
**Task Mechanism**: Dynamically reprioritize subtasks based on causal opportunity signals.
**Key Results**: Reports improved long-horizon success and efficiency on sparse Minecraft tasks.
**Reproduction Difficulty**: Medium; causal graph and scheduler can be approximated without retraining.
**Borrowable Points**:
  - Store why an event mattered, not just that it happened.
  - Reorder accepted tasks when a causal opportunity appears.
  - Use failure causality to revise low-level controller behavior.
**Current Singularity Mapping**:
  - `ExperienceRecord.causal` stores which/why.
  - `TaskSystem.opportunity_triggers` and `get_next_task(world_state)` already support opportunistic reordering.
  - `CausalEventIndex` now stores action transitions keyed by subject, action, outcome, tags, and compact before/after evidence.
  - Live `Agent` actions and offline `SkillExtractor.extract_experience_atoms()` both populate causal events.
  - `Agent` now enriches scheduling state with causal tags/events, and `TaskSystem` gives secondary opportunity credit to matching causal triggers.
  - `TaskSystem(use_causal_opportunities=False)` and `scheduling-ablation` provide a direct-only versus causal-enabled comparison.
  - `scheduling-ablation --session-log` replays real session transitions, including successful and failed actions, into causal scheduling cases.
  - `CausalEvent.value_score` filters low-value transitions before they become scheduling candidates.
  - Repeated action/subject/outcome transitions now aggregate into compact summaries with repeat counts, merged causal tags, max/average value scores, and representative evidence.
  - Stable repeated successful summaries now become reviewable skill candidates through `SkillExtractor.extract_causal_skill_candidates()` and benchmark ingestion.
  - Repeated high-value failures now pair with nearby successful correction sequences and become reviewable `failure_correction_summary` skill candidates.
  - Reviewed causal/correction skills now influence online behavior through planner hints and failure-triggered correction sequences loaded from `Config.skill_dir`.
  - Online policy-skill interventions are now counted in session summaries and benchmark results so runs can report hints, correction attempts, actions, successes, failures, success rate, and involved skills.
  - `policy-skill-ablation` now compares reviewed skills disabled versus enabled on deterministic online correction cases.
  - `policy-skill-ablation --skill-storage-path` now loads approved `failure_correction_skill` entries from `custom_skills.jsonl` and builds multi-case disabled/enabled tests from them.
  - `benchmark --policy-skill-ablation` now compares reviewed policy skills disabled versus enabled on the selected live M1/M2/all benchmark suite.
