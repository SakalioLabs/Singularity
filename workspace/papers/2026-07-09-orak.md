# Paper Card: Orak

**Title**: Orak: A Foundational Benchmark for Training and Evaluating LLM Agents on Diverse Video Games
**Source**: https://arxiv.org/abs/2506.03610
**Date**: 2025-06-04; active versions in 2026
**Type**: Multi-game LLM-agent benchmark and training dataset
**Core Problem**: Game-agent benchmarks often test isolated capabilities or narrow game families, making it hard to compare LLM models, perception choices, memory, planning, and other agentic modules under consistent conditions.
**Core Method**:
  - Evaluates agents across 12 popular video games spanning major genres.
  - Uses a plug-and-play interface based on Model Context Protocol so agentic modules can be studied systematically.
  - Includes general score leaderboards, battle arenas, visual-input studies, strategy analyses, and fine-tuning data from gameplay trajectories.
**Why It Matters for Minecraft**:
  - Singularity already has many optional modules: visual action grounding, memory quality feedback, plan cache, action-value scoring, mixed-policy patches, and runtime profiles.
  - Orak reinforces that these modules need comparable on/off reports and suite-level readiness checks instead of one-off success anecdotes.
  - The MCP-style plug-and-play framing maps well to our runtime-profile and preflight gates for M1/M2/M7.
**Singularity Adaptation**:
  - Treat new modules as experiment units with offline report, ablation/runtime report, gate, runtime-profile packaging, and suite preflight.
  - The new `plan-cache-runtime-report` and `plan-cache-gate` follow this pattern: cache reuse is measured as a module before it is allowed into live profiles.
  - `agent-module-comparison-report` now publishes comparable module summaries: hit rates, verifier rejects, action failures, completion deltas, and candidate active modules across baseline/candidate logs.
**Next Action**: Run `agent-module-comparison-report` on fresh live M1/M2/M7 baseline and runtime-profile candidate logs, then only promote candidates whose dedicated gates also pass.
