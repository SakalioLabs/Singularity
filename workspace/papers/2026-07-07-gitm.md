# Paper Card: GITM

**Title**: GITM: Ghost in the Minecraft — Generally Capable Agents for Minecraft with Text-based Knowledge and Memory
**Source**: https://arxiv.org/abs/2305.17209
**Date**: 2023
**Authors**: Zihao Wang, Shaofei Cai, Anji Liu, Xiaojian Ma, Yitao Liang (Peking University)
**Type**: Paper + Code
**Core Problem**: Can a text-only LLM solve long-horizon Minecraft tasks without visual input?
**Core Method**: Text-only LLM + 300+ structured text actions + hierarchical planning
**System Architecture**:
  - Subgoal Decomposer: Breaks high-level goals into subgoals
  - Planning Controller: Manages subgoal execution order
  - Action Controller: Translates subgoals into atomic text actions
  - Environment Interface: Reads/writes Minecraft state as text
**Input**: Text-based environment state (block info, inventory, position)
**Output**: Text actions from 300+ action vocabulary
**Action Space**: 300+ structured text actions (goto, mine, place, craft, smelt, etc.)
**Memory Mechanism**: Text-based knowledge base + working memory
**Task Mechanism**: Hierarchical subgoal decomposition
**Skill Mechanism**: Fixed action vocabulary (no learning)
**Evaluation Tasks**: Obtain diamond (long-horizon survival task)
**Key Results**:
  - 67.5% success rate on diamond obtainment task
  - Outperformed RL baselines (MineDojo RL: 0% success)
  - Demonstrated text-only LLM viability for complex tasks
  - No visual input needed for many tasks
**Failure Cases**:
  - Fixed action vocabulary limits flexibility
  - No skill learning or transfer
  - Relies heavily on structured state representation
  - Struggles with dynamic environments
**Dependencies**: Custom Minecraft environment, OpenAI API
**Is Open Source**: Yes - https://github.com/OpenGVLab/GITM
**Reproduction Difficulty**: Medium (custom environment)
**Borrowable Points**:
  - Structured text action vocabulary design (300+ actions)
  - Hierarchical subgoal decomposition
  - Text-only state representation proves viable
**Cannot Directly Adopt Because**:
  - Custom environment interface (not Mineflayer)
  - Fixed action vocabulary (no extensibility)
**Suggestion for This Project**:
  - Adopt structured action vocabulary concept
  - Use text-based state representation for LLM input
  - Implement hierarchical planning on top of Mineflayer
**Next Action**: Design action vocabulary inspired by GITM for our Mineflayer bridge
